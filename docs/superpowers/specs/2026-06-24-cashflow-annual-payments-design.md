# 資金繰り（年間の大型支払い登録＋資金残高の推移）設計

作成日：2026-06-24
対象ブランチ：feature/sales-management-mvp

## 1. 目的・背景

研修会社では、消費税（中間・確定で年2回）、社会保険料、労働保険料、法人税などの
**年間を通して金額が決まっている大きな支払い**がある。金額が大きいため、
「この月にこの支払いがある」と事前に登録しておき、**手元資金の残高がどう推移するか**を
見える化したい。

本機能で実現すること：
1. **大型支払いの登録枠**：項目（行）× 12ヶ月（4月→3月）の表に金額を入力。
   定番項目は初期投入し、ユーザーが項目を追加・名称変更・削除できる。
2. **資金残高の推移**：「入金 − 支払い」を月次で積み上げ、期首残高を起点に累計残高を表示。
   入金側は「請求ベース／入金ベース」をトグルで切替。

## 2. 配置

新規ページ **「資金繰り」（`/cashflow`）** を追加する（既存画面は変更しない）。
売上の集計（年間売上管理表）とは関心事が違う（お金がいつ出ていくか）ため独立画面とする。
メニューは既存「入金」グループ内に追加。

> 命名上の注意：既存に `payments` ルーター・`/payments`（入金管理）ページがあるため、
> 本機能のルーター/ページは衝突回避で **`cashflow`** とする。

### 3.2 状態

- **loading**：既存 `<Loading/>`
- **empty**：データ未入力でも表は0で全枠表示（入力が主目的）。推移は期首残高のみの水平線
- **error**：既存 `<ErrorState/>`
- **saving**：保存中インジケータ＋完了トースト（既存パターンに準拠）

### 3.3 操作フロー

1. 年度を選ぶ（FiscalYearContext を流用）→ 既存データ読込
2. 期首残高を入力して保存
3. 大型支払い表のセルに金額入力 →「まとめて保存」
4. 下段の推移が反映。請求/入金ベースをトグルで見比べ
5. 項目の追加・名称変更・削除（金額が入った項目の削除は確認ダイアログ）

## 4. データモデル

### 4.1 新規テーブル `PaymentItem`（大型支払いの項目マスタ）

| カラム | 型 | 説明 |
|---|---|---|
| id | int PK | |
| name | str | 項目名（例：消費税） |
| sort_order | int | 表示順 |
| active | bool=True | 削除は論理削除（過去金額を残す） |
| created_at | datetime | |

初回起動時に定番6項目をシード（無ければ投入）：
消費税 / 社会保険料 / 労働保険料 / 法人税等 / 住民税・事業税 / 源泉所得税。
年度をまたいで共有（毎年同じ行を使い、金額のみ年度別）。

### 4.2 新規テーブル `PaymentSchedule`（月別の金額）

| カラム | 型 | 説明 |
|---|---|---|
| item_id | int PK | `PaymentItem.id` |
| ym | str PK | "YYYY-MM" |
| amount | int=0 | その月の支払額 |
| updated_at | datetime | |

複合主キー (item_id, ym)。年度別は ym で自然に分かれる（既存 `MonthlyFixedCost` と同じ発想）。

### 4.3 既存テーブル変更 `Setting`

| 追加カラム | 型 | 説明 |
|---|---|---|
| opening_balance | int=0 | 年度開始時点の手元資金 |

`_run_migrations()` に `ALTER TABLE setting ADD COLUMN opening_balance INTEGER DEFAULT 0` を追記。
新規2テーブルは `create_all` で自動作成。

### 4.4 戻し方

- 新規2テーブルは未使用なら無害（drop可）
- `Setting` の追加列はデフォルト0で既存挙動に影響なし

## 5. 計算ロジック

12ヶ月（4月→3月の年度順、index 0..11）それぞれで計算する。

### 5.1 入金（プラス側）— トグルで切替

- **請求ベース**：案件の `billing` を売上月（revenue_month）に計上（年間売上管理表と同じ集計）
- **入金ベース**：`payment_status='paid'` の `paid_amount` を入金日（paid_on）の月に計上。
  入金日が未入力の入金済みは当月に寄せず、合計欄に「入金日未入力 ¥X」と注記する

### 5.2 支払い（マイナス側）

- **大型支払い**：`PaymentSchedule` の月別合計（本機能の主役）
- **月次固定費**：`_month_fixed_cost()`（既存ヘルパー流用、未設定月は年度設定へフォールバック）
- **原価（講師料）**：`direct_cost`（未設定なら `instructor_fee`）を売上月に計上。
  入金ベースでも支払時期データが無いため売上月で近似（画面に注記）

### 5.3 残高

```
月次収支[i] = 入金[i] − (大型支払い[i] + 固定費[i] + 原価[i])
累計残高[i] = 期首残高 + Σ 月次収支[0..i]
```

累計残高がマイナスの月は赤字表示（資金ショート警告）。

## 6. API

| メソッド・パス | 説明 |
|---|---|
| `GET /api/cashflow?fiscal_year=YYYY&basis=billing\|paid` | 推移データ（labels / 入金 / 固定費 / 原価 / 大型支払い / 月次収支 / 累計残高 / 期首残高 / 入金日未入力額） |
| `GET /api/cashflow/items` | 大型支払いの項目一覧（active のみ、sort_order順） |
| `POST /api/cashflow/items` | 項目追加（name） |
| `PATCH /api/cashflow/items/{id}` | 名称変更・並び替え |
| `DELETE /api/cashflow/items/{id}` | 論理削除（active=False） |
| `GET /api/cashflow/schedule?fiscal_year=YYYY` | 項目×月の金額マトリクス |
| `PUT /api/cashflow/schedule?fiscal_year=YYYY` | 金額一括保存（item_id × ym × amount） |
| `GET/PUT 期首残高` | `routers/settings.py` に `opening_balance` を追加して取得/保存 |

ルーターは `routers/cashflow.py`（新規）に集約。既存の `_deals_in_fy` / `_month_fixed_cost` /
`calc.monthly_buckets` を流用（必要なら共通関数を `calc` か `summary` から参照）。
admin 権限前提（既存 summary 同様 `require_admin`）。

## 9. 受け入れ条件（Before/After）

| 項目 | Before | After |
|---|---|---|
| 大型支払いの登録 | 登録枠がない | 項目×12ヶ月の表に金額入力・項目追加/削除でき、保存される |
| 期首残高 | なし | 年度ごとに入力・保存できる |
| 資金残高の推移 | 見えない | 月次の入金−支払いと累計残高がグラフ＋表で見える |
| 入金基準 | — | 請求ベース／入金ベースを切替表示できる |
| 資金ショート | 気づけない | 累計残高がマイナスの月を赤字で警告 |

## 10. 非対象（YAGNI）

- 大型支払いの自動計算（消費税額の自動算出など）は対象外。金額は手入力。
- 実際の銀行残高連携・入出金明細取込は対象外。
- 原価・固定費の「実際の支払時期」管理は対象外（売上月/月次で近似）。
